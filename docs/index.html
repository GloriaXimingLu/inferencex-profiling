The short version

  • The cost of serving a large model is basically GPU time. The software that runs the model has dozens of low-level settings that can swing cost by ~2×, but they interact in non-obvious ways and are tedious to tune by hand.
  • We pointed an AI agent at the problem. It proposes a server configuration, we measure the true cost on a real GPU, and it evolves better configurations over many rounds — automatically.
  • It found a setup that serves the same model on the same GPU at 43% lower cost per token (about 1.8× more tokens per second) — and discovered it on its own.
  • We validated it on real workloads. Replaying actual multi-step agent conversations, throughput rose +9 to +70% when the system is busy (where cost matters most), and was roughly flat when lightly loaded.

1The problem: serving cost is GPU cost

We run a 120-billion-parameter open model (GPT-OSS-120B) on a single high-end GPU (an NVIDIA H200), using vLLM — the open-source engine that actually serves the model to users. Our bill is essentially GPU-hours: the more user output we produce per GPU-second, the lower our cost per token.

The catch: vLLM has dozens of launch settings — how many requests to batch together, how to lay out memory, which math routines to use, and so on. They interact in surprising ways and can swing throughput 2× or more. Finding the best combination by hand is slow and easy to get wrong.

In plain terms: it's like tuning an engine. The same car (model) and same fuel (GPU) go much further on the right settings — but there are too many dials to try by hand, and the best combination isn't obvious.

The goal: find the configuration that serves the most tokens per GPU-second — the cheapest — without making the service too slow for users.

2What we did: let an agent search for us

We used OpenEvolve, an evolutionary AI agent. It runs guided trial-and-error: propose a server configuration, launch the real model on a real GPU and measure the true cost per token, then learn from each result to propose better configurations — over many rounds.

Two things make this safe and trustworthy:

  • It only changes how the model is served — never the model itself. Output quality is untouched, so the savings carry no quality risk.
  • It's scored on real measurements, not estimates. Every configuration runs on an actual GPU under load, so a "win" is a real win.

Notably, the agent discovered the winning configuration on its own — we did not hand it the answer.

3The results

3.1 — 43% lower cost per token

The configuration the agent found serves the same model on the same GPU at 43% lower cost per token — equivalently, about 1.8× more tokens per GPU-second. The savings came from three changes the agent chose: a faster math routine for this model, a more compact memory format, and larger processing batches.

For the technically curious: the actual settings the agent chose
  • Triton MoE kernel (moe_backend=triton) — a faster mixture-of-experts math routine for this model on H200, replacing the slower default.
  • FP8 KV cache (kv_cache_dtype=fp8) — a more compact format for the attention cache, freeing memory for bigger batches.
  • Larger batches (max_num_seqs=768, max_num_batched_tokens=36864) — process many more requests together.
  • Speculative decoding (EAGLE3, num_speculative_tokens=2) plus a streaming tweak (stream_interval=10).

3.2 — Beats our previous setup everywhere; best-in-class at peak

This chart maps the core trade-off in serving: per-user speed (how fast one user's reply streams — right is snappier) versus total throughput (how many users we serve per GPU — up is cheaper). Our auto-tuned config (green) sits above our previous setup (gray) at every load level. At the highest-throughput operating point — where serving cost is lowest — it matches and slightly exceeds a leading commercial serving stack (orange). There's still headroom versus that stack in the middle of the range, which is where we'd look to improve next.

3.3 — The win holds on real agent traffic

Lab benchmarks can flatter a result, so we validated on realistic traffic: replaying actual multi-step agent conversations (τ-bench — airline booking, banking Q&A, and similar tool-using dialogues). Under heavy load — many users at once, exactly when GPU cost bites — throughput rose +9 to +70% (three of the four workloads gained ~65%). Under light load it was flat-to-slightly-lower — which doesn't affect cost: a lightly-loaded GPU is underused either way, so there's no throughput to win there.

4The honest trade-off

The cheapest configuration hits its throughput by processing many requests together, which slightly slows any single user's reply. That's the right call when the GPU is busy serving many users (the costly case), and it matters less when traffic is light.

For latency-sensitive deployments, the agent also found a "deployable" variant that delivers +16% more throughput while staying within strict responsiveness limits. Net: we now have a menu of configurations along the cost-vs-speed trade-off — all better than where we started.

5Bottom line & what's next

This is a concrete, no-risk efficiency win: up to 43% lower cost per token on the same model and hardware, validated on production-style traffic. Two ways to capture it:

→ High-traffic & batch serving
Adopt the throughput-max config. Biggest savings exactly where our load (and cost) is highest. Drop-in: no model change.
→ Latency-sensitive endpoints
Use the deployable variant: +16% throughput while staying within strict responsiveness limits.

And the tuning agent is reusable: when we change models or GPUs, it re-tunes automatically — so this is a repeatable capability, not a one-off result. Next step: close the remaining mid-range gap versus the commercial reference, and run the agent against our next target model.

Glossary

vLLM
The open-source software engine that runs our model and answers user requests. Its launch settings are what we tuned.
Token
A chunk of text (roughly ¾ of a word). Models read and write in tokens; we translate GPU time into cost per token.
Throughput
Total output tokens produced per second across all users. Higher = more work per GPU = lower cost.
Per-user speed
How fast one user's reply streams out. Trade-off: serving more users at once can slow each individual reply.
Concurrency / load
How many users the system handles at the same time. "Heavy load" = many at once.
Cost per token
Our key metric: GPU cost divided by tokens produced. Lower is the whole goal.
Server config
The launch settings (batch sizes, memory format, math routines, …) the agent tunes. The model is never changed.
OpenEvolve
The evolutionary AI agent we used to search the settings automatically.
τ-bench
A public benchmark of realistic multi-step agent conversations (booking, support, Q&A) used to validate on production-style traffic.

Methodology in one line: all numbers measured on a single NVIDIA H200 serving GPT-OSS-120B via vLLM; throughput compared on matched workloads; cost = GPU-time ÷ output tokens. The agent optimized only launch configuration, never the model.
