Auto-tuning the vLLM serving configuration for GPT-OSS-120B

An evolutionary search agent (OpenEvolve) tuned the vLLM launch configuration for GPT-OSS-120B on a single NVIDIA H200, with the objective of minimizing cost per output token. This report presents the measured results — cost, throughput, and latency — on a synthetic benchmark and on replayed τ-bench traffic. The model weights were not modified.

Summary of findings

  • On the synthetic benchmark (8k-token input, 1k-token output) at concurrency 256, the configuration found by the optimization reduced cost per output token from $1.83 to $1.05 per 1M tokens (−43%) and raised output throughput from 1,617 to 2,943 tokens/sec (+82%), at a GPU rate of $3.50 per H200-hour.
  • The optimization targeted concurrency 256, so the configuration is tuned for high concurrency. A throughput sweep confirms it beats the baseline at 8 of 9 concurrency levels (1–256) and exceeds a commercial reference (Fireworks) at concurrency 128 and 256.
  • On replayed τ-bench agent traffic the same pattern holds: throughput changed by +9% to +70% at concurrency 64 (airline) and by −18% to +3% at concurrency 4 (banking) — large gains at high concurrency, roughly flat at low concurrency.
  • A configuration for a low-concurrency, latency-sensitive regime is obtained by re-running the search at that concurrency. Done at concurrency 16, this produced a latency-constrained configuration with +16% throughput over the baseline within p99 limits of 5,000 ms / 25 ms (measured 4,873 ms / 20.6 ms).
  • The optimization modified only the vLLM launch configuration; model weights were unchanged.

1Setup

  • Model: GPT-OSS-120B (120B parameters, MXFP4 weights).
  • Hardware: one NVIDIA H200 GPU.
  • Serving engine: vLLM v0.22 — the open-source engine that runs the model and answers requests. Its launch configuration (the settings used at startup) is what was tuned.
  • Cost metric: cost per 1M output tokens = (wall-clock GPU-hours × GPU hourly rate) ÷ output tokens. GPU rate: $3.50 per H200-hour (1 GPU, so GPU-hours = wall-clock hours). Because each request processes an 8k-token input before generating output, this is the full serving cost — including input-processing time — divided across the output tokens; it is therefore not simply the inverse of output throughput. The percentage change between configurations is independent of the rate.
  • Objective: minimize cost per output token at a target concurrency, subject to latency limits.

The launch configuration exposes on the order of 15 settings — batch sizes, key/value-cache numeric format, attention and mixture-of-experts compute kernels, and speculative-decoding depth — that affect throughput and latency. These were the search space.

2Method

OpenEvolve is an evolutionary search agent. Each round proceeds as follows, at a chosen target concurrency:

4 · Evolve
results fed back; new configs proposed
  • Each candidate was scored on measured cost per token from a live GPU under load, not on estimates.
  • The search modified only launch settings (batch sizes, cache format, kernel selection, speculative-decoding depth). Model weights were unchanged.
  • The reported configuration was produced by the search; it was not specified manually.
  • The search runs at a chosen concurrency. The throughput-max configuration was found at concurrency 256; the latency-constrained configuration (§4–5) was found by re-running the search at concurrency 16.

3Results

3.1 — Cost and throughput at concurrency 256 (synthetic benchmark)

On the synthetic benchmark (8k-token input, 1k-token output), the optimization — run at concurrency 256 — found a configuration that reduced cost per output token from $1.83 to $1.05 per 1M tokens (−43%) (at $3.50 per H200-hour) and raised output throughput from 1,617 to 2,943 tokens/sec (+82%). The cost figure includes the time to process each request's 8k-token input.

Settings the search selected (vs. baseline)
  • Mixture-of-experts kernel: moe_backend=triton (baseline used the Marlin kernel).
  • Key/value-cache format: kv_cache_dtype=fp8 (more compact than the baseline format).
  • Batch sizes: max_num_seqs=768, max_num_batched_tokens=36864 (baseline: 256 / 8192).
  • Speculative decoding: EAGLE3, num_speculative_tokens=2; streaming interval stream_interval=10.

3.2 — Throughput across concurrency levels

The chart plots per-user output speed (tokens/sec for a single user) against total throughput (tokens/sec across all users) at concurrency levels 1–256, from a sweep separate from the optimization run. The configuration's throughput exceeded the baseline at 8 of 9 measured levels; at concurrency 4 the baseline was higher (465 vs. 520 tokens/sec). At concurrency 128 and 256 it exceeded the commercial reference (2,335 vs. 2,180 and 2,657 vs. 2,300 tokens/sec); at intermediate per-user speeds the commercial reference produced higher throughput than both. The gap over the baseline is largest at high concurrency, consistent with the optimization having targeted concurrency 256.

3.3 — Throughput on replayed τ-bench traffic

τ-bench multi-turn agent conversations (airline and banking domains) were replayed against both configurations at matched request counts and matched concurrency. At concurrency 64 (airline), throughput changed by +9%, +63%, +69%, +70% across the four source workloads. At concurrency 4 (banking), throughput changed by −18%, +3%, −12% across three workloads (a fourth had no valid baseline and is omitted).

The result depends on concurrency, by design. The throughput-max configuration was found by optimizing at concurrency 256, so it is tuned for high-concurrency serving. The τ-bench results above follow directly: large gains at concurrency 64 (airline), roughly flat at concurrency 4 (banking). A configuration for a low-concurrency, latency-sensitive regime is obtained by re-running the same search at that concurrency — the latency-constrained configuration (§4–5) was produced this way at concurrency 16, and other concurrency regimes can be targeted the same way.

4Latency

Per-request latency depends on the concurrency the server runs at. At the maximum measured concurrency (256), where throughput is highest, p99 time-to-first-token is high for both configurations because many requests are queued: 113 s for the baseline and 39 s for the throughput-max configuration. p99 inter-token latency (the time between successive output tokens, also called TPOT) was 144 ms for the baseline and 157 ms for the throughput-max configuration. The high time-to-first-token reflects the request-queue depth at this concurrency, not a property of either configuration; at lower concurrency it is much smaller.

To obtain a configuration suitable for latency-sensitive use, the search was re-run at concurrency 16 under explicit latency limits (p99 time-to-first-token ≤ 5,000 ms, p99 inter-token latency ≤ 25 ms). It produced +16% throughput over the baseline while meeting the limits — measured p99 time-to-first-token 4,873 ms and p99 inter-token latency 20.6 ms.

Glossary
Throughput
Output tokens produced per second across all users. Higher throughput corresponds to lower cost per token.
Per-user speed
Output tokens per second for a single user.
Concurrency
Number of requests handled at the same time (reported as 1–256 here). The optimization runs at a chosen concurrency.
Time-to-first-token (TTFT)
Delay before a user receives the first output token. "p99" = the 99th-percentile (near-worst-case) value.
Inter-token latency (TPOT)
Time between successive output tokens for one user. Also called time-per-output-token (TPOT); the two terms refer to the same metric here.
Launch configuration
Startup settings (batch sizes, cache format, kernels, speculative-decoding depth). The model is not part of it.
MoE kernel
The compute routine for the model's mixture-of-experts layers (e.g., Marlin or Triton).
KV cache
Memory holding intermediate state during generation; its numeric format (e.g., fp8) trades precision for capacity.
Speculative decoding
Generating several draft tokens per step and verifying them, to raise throughput.
OpenEvolve
The evolutionary search agent used to tune the configuration.
Fireworks
A commercial LLM-serving provider, used here as an external throughput reference.
τ-bench
A public benchmark of multi-turn agent conversations (airline, banking, etc.), replayed here as a realistic traffic pattern.

Methodology. Model: GPT-OSS-120B (MXFP4) on 1× NVIDIA H200, vLLM v0.22. Synthetic benchmark: 8k-token input / 1k-token output, concurrency 1–256. The optimization (OpenEvolve) was run at concurrency 256 for the throughput-max configuration and concurrency 16 for the latency-constrained configuration. τ-bench replay: recorded multi-turn agent conversations (airline, banking) replayed with synthetic tokens at matched request counts and concurrency to the baseline; throughput computed from wall-clock time over forced-length outputs (server-side token counters were not used, as the streaming and speculative-decoding settings affect them). Cost per 1M output tokens = (wall_s ÷ 3600 × num_GPUs × $3.50/GPU-hr) ÷ output tokens × 1,000,000, on 1 GPU; it includes input-prefill time. The search optimized only the vLLM launch configuration; model weights were unchanged.
